## Your First Session

Get started with bdg in three simple steps:

<Steps>
  <Step title="Start a debugging session">
    Launch Chrome and connect to any website:
    
    ```bash
    bdg example.com
    ```
    
    This will:
    - Launch Chrome in the background
    - Navigate to the URL
    - Establish a CDP connection
    - Start collecting telemetry data
    
    <Note>
      The session runs in the background. Your terminal is free for other commands.
    </Note>
  </Step>
  
  <Step title="Inspect the page">
    Query DOM elements and check what's on the page:
    
    ```bash
    # Find all buttons
    bdg dom query "button"
    ```
    
    <CodeGroup>
    
    ```bash Query buttons
    bdg dom query "button"
    ```
    
    ```bash Check status
    bdg status
    ```
    
    ```bash View network activity
    bdg peek --network
    ```
    
    ```bash Check console messages
    bdg console
    ```
    
    </CodeGroup>
  </Step>
  
  <Step title="Stop the session">
    When you're done, stop the session:
    
    ```bash
    bdg stop
    ```
    
    This will:
    - Close Chrome
    - Save collected data to `~/.bdg/session.json`
    - Clean up session files
  </Step>
</Steps>

## Common Commands

Here are the most frequently used commands to get you started:

### Session Management

<CodeGroup>

```bash Start session
bdg localhost:3000
```

```bash Check status
bdg status --verbose
```

```bash Stop session
bdg stop
```

```bash Clean up stale sessions
bdg cleanup --force
```

</CodeGroup>

### DOM Inspection

<CodeGroup>

```bash Query elements
bdg dom query "button.submit"
```

```bash Get element details
bdg dom get "h1"
```

```bash Evaluate JavaScript
bdg dom eval "document.title"
```

```bash Check accessibility tree
bdg dom a11y tree
```

</CodeGroup>

### Network Monitoring

<CodeGroup>

```bash List requests
bdg network list
```

```bash Filter by status
bdg network list --filter "status-code:>=400"
```

```bash Export HAR file
bdg network har capture.har
```

```bash Check headers
bdg network headers
```

</CodeGroup>

### Live Monitoring

<CodeGroup>

```bash Preview recent data
bdg peek --last 20
```

```bash Stream updates
bdg tail --network
```

```bash Filter by type
bdg peek --type XHR,Fetch
```

```bash Console messages
bdg console --follow
```

</CodeGroup>

## Real-World Example

Let's debug a real website and inspect its network activity:

<Steps>
  <Step title="Start session with a popular site">
    ```bash
    bdg https://news.ycombinator.com
    ```
    
    Wait a moment for the page to load completely.
  </Step>
  
  <Step title="Check what loaded">
    View network requests:
    
    ```bash
    bdg peek --network --last 50
    ```
    
    This shows the most recent network requests with status codes and URLs.
  </Step>
  
  <Step title="Find all links">
    Query for links on the page:
    
    ```bash
    bdg dom query "a.storylink"
    ```
    
    You'll see a list of all story links with their text content.
  </Step>
  
  <Step title="Extract article titles">
    Use JavaScript evaluation to get titles:
    
    ```bash
    bdg dom eval "Array.from(document.querySelectorAll('a.storylink')).map(a => a.textContent)"
    ```
    
    Returns a JSON array of all article titles.
  </Step>
  
  <Step title="Check for errors">
    Look for console errors or warnings:
    
    ```bash
    bdg console
    ```
    
    Shows any JavaScript errors or console messages.
  </Step>
  
  <Step title="Export network data">
    Save network activity for analysis:
    
    ```bash
    bdg network har hn-capture.har
    ```
    
    Creates a HAR file you can open in Chrome DevTools.
  </Step>
  
  <Step title="Clean up">
    ```bash
    bdg stop
    ```
    
    Session data saved to `~/.bdg/session.json`.
  </Step>
</Steps>

## Advanced Options

### Headless Mode

Run Chrome without a visible window:

```bash
bdg example.com --headless
```

Perfect for CI/CD pipelines and automated testing.

### Custom Chrome Flags

Pass flags to Chrome for special scenarios:

<CodeGroup>

```bash Ignore SSL errors
bdg https://localhost:5173 --chrome-flags="--ignore-certificate-errors"
```

```bash Disable CORS
bdg https://localhost:3000 --chrome-flags="--disable-web-security"
```

```bash Custom window size
bdg example.com --chrome-flags="--window-size=1920,1080"
```

</CodeGroup>

### Connect to Existing Chrome

Connect to a Chrome instance you already have running:

<Steps>
  <Step title="Launch Chrome with debugging enabled">
    ```bash
    # macOS
    /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
      --remote-debugging-port=9222
    
    # Linux
    google-chrome --remote-debugging-port=9222
    ```
  </Step>
  
  <Step title="Connect bdg to it">
    ```bash
    bdg --chrome-ws-url ws://localhost:9222/devtools/browser/<id>
    ```
    
    Get the WebSocket URL from `http://localhost:9222/json`.
  </Step>
</Steps>

## Understanding Output

### JSON Output

All commands support `--json` for programmatic use:

```bash
bdg status --json
```

```json
{
  "version": "0.7.2",
  "success": true,
  "data": {
    "active": true,
    "pid": 12345,
    "url": "https://example.com",
    "uptime": 42000
  }
}
```

### Exit Codes

bdg uses semantic exit codes for error handling:

- **0** — Success
- **80-99** — User errors (invalid arguments, resources not found)
- **100-119** — System errors (timeouts, connection failures)

Check exit codes in scripts:

```bash
if bdg dom query "button"; then
  echo "Found buttons"
else
  echo "No buttons found (exit code: $?)"
fi
```

<Note>
  See the [Exit Codes](/advanced/exit-codes) reference for complete details.
</Note>

## Working with Pipes

bdg works seamlessly with Unix pipes and jq:

```bash
# Extract specific data with jq
bdg peek --json | jq '.data.preview.data.network[] | select(.status >= 400)'

# Count requests by status
bdg network list --json | jq '[.data[].status] | group_by(.) | map({status: .[0], count: length})'

# Get all console errors
bdg console --json | jq '.data.messages[] | select(.level == "error")'
```
